CLAUDE.md — 보안 자동 체크 파일
이 파일 하나면, AI가 코드를 만들 때마다 보안을 자동으로 확인해줍니다
🤖 비유: 자동 맞춤법 검사기와 같습니다. 한번 설정해두면 글을 쓸 때마다 알아서 틀린 부분을 찾아주죠? CLAUDE.md도 마찬가지로, 코드를 만들 때마다 보안 문제를 자동으로 찾아줍니다.
어떻게 쓰나요? (3가지 방법)
본인의 AI 사용 환경에 맞게 선택하세요. 해당되는 방법을 모두 설정하면 가장 안전합니다.
📁 방법 1 — 프로젝트 폴더에 넣기 (자동 체크)
대상: Claude Code로 코딩하는 직원
CLAUDE.md를 다운로드 → 프로젝트 폴더에 넣기 → 끝! 이후 자동으로 보안 체크됩니다.
다운로드 후 프로젝트 폴더 최상위에 넣으세요.
💬 방법 2 — "보안 확인해줘"로 언제든 호출 (Memory 등록)
대상: Claude Code 사용하는 모든 직원
별도 유료 기능이 아닙니다. Claude Code를 쓰는 사람이면 누구나 가능합니다.
최초 1회만 등록하면 이후 자동으로 기억이 유지됩니다. 매번 다시 입력할 필요 없습니다.
복사 후 Claude Code 대화창( > 표시)에 Ctrl+V → Enter. "기억했습니다" 응답이 오면 완료.
🌐 방법 3 — Claude 웹/앱 프로젝트에 등록 (팀 전체 공유)
대상: claude.ai (웹사이트/앱)를 사용하는 직원
claude.ai 접속 → 프로젝트 생성 → 지식 추가에 방법 1에서 다운받은 CLAUDE.md를 드래그해서 넣기 → 끝!
프로젝트 링크를 팀원에게 공유하면 팀 전체가 같은 보안 규칙을 적용받습니다.
💡 어떤 방법을 선택하나요?
• Claude Code로 코딩하는 직원 → 방법 1 + 2 둘 다 설정
• Claude 웹/앱으로 질문하는 직원 → 방법 3 설정
• 둘 다 쓰는 직원 → 1 + 2 + 3 모두 설정

🔎 두번째
왜 이런 규칙이 필요한가요?
CLAUDE.md에 들어있는 보안 규칙들, 비유와 실제 코드로 쉽게 설명합니다.
01
비밀번호에 랜덤 값을 섞어야 해요
같은 비밀번호도 매번 다르게 저장하는 방법
🔒 왜? 비밀번호를 그대로 저장하면, 누군가 DB를 열었을 때 모든 비밀번호가 그대로 보입니다. 해시(암호화)로 저장하면 의미 없는 암호문만 보여서 원래 비밀번호를 알 수 없습니다.

추가로: 같은 비밀번호는 항상 같은 암호 결과가 나와서, 해커가 역으로 맞출 수 있어요. 랜덤 값(Salt)을 섞으면 같은 비밀번호라도 매번 다른 결과가 나와 해킹이 불가능해집니다.
✅ DO
SHA-256 + Salt 해시로 변환 후 저장
비교는 해시 vs 해시로만
비밀번호 원문은 어디에도 없게
❌ DON'T
HTML/JS에 비밀번호 하드코딩
Firebase에 평문 비밀번호 저장
console.log()로 비밀번호 출력
▼ 코드 예시 보기
// SHA-256 해시 변환 함수 async function sha256(str) { const buf = await crypto.subtle.digest( 'SHA-256', new TextEncoder().encode(str) ); return Array.from(new Uint8Array(buf)) .map(b => b.toString(16).padStart(2,'0')) .join(''); } // 저장: 해시만 저장 const hash = await sha256('비밀번호'); db.collection('passwords').doc('teamA') .set({ hash: hash }); // 검증: 해시끼리 비교 const inputHash = await sha256(입력값); if (inputHash === 저장된해시) { // 로그인 성공 }
02
데이터마다 출입증을 따로 만들어야 해요
Firebase 보안 규칙 — 누가 어떤 데이터에 접근할 수 있는지
🏢 왜? 우리 사이트의 소스코드를 열면 Firebase 접속 정보가 보입니다. 보안 규칙이 없으면 이 정보만으로 누구든 우리 데이터를 몽땅 지워버릴 수 있습니다.

비유: 회사 건물에 출입증이 있죠? 1층 직원은 1층만, 3층 직원은 3층만. Firebase도 마찬가지로 서비스 A의 데이터는 서비스 A만, 서비스 B의 데이터는 서비스 B만 접근하도록 각각 다른 출입증(보안 규칙)을 설정해야 합니다.
✅ DO
내 서비스 전용 컬렉션만 허용
나머지는 전부 차단
여러 서비스는 접두사로 구분
❌ DON'T
allow: true로 전체 허용
접두사 없이 "users" 등 사용
보안 규칙 미설정 상태로 배포
▼ 코드 예시 보기
rules_version = '2'; service cloud.firestore { match /databases/{database}/documents { // 내 서비스 전용 컬렉션만 허용 match /my-service-links/{doc} { allow read, write: if true; } match /my-service-settings/{doc} { allow read, write: if true; } // 나머지는 전부 차단 match /{document=**} { allow read, write: if false; } } }
03
데이터 이름을 쉽게 못 찾게 바꿔야 해요
데이터 이름만으로 뭔지 추측할 수 없게 만들기
🔎 왜? 금고에 "비밀번호 보관함"이라고 크게 써 있으면 도둑이 제일 먼저 노리겠죠? 데이터 이름도 마찬가지입니다. "users", "passwords" 처럼 누구나 추측 가능한 이름을 쓰면 해커가 바로 찾아갑니다.
❌ users, passwords, data, settings, admin
✅ portal-pw, hr-employees, kpi-scores
규칙: 서비스명-기능 형태의 접두사를 붙이세요. 외부에서 추측하기 어렵고, 같은 Firebase에 여러 서비스를 운영할 때 데이터가 섞이는 것도 방지합니다.
04
AI가 만든 코드, 그대로 쓰면 안 돼요
이것도 CLAUDE.md가 자동으로 체크해줍니다
👷 비유: AI가 만든 코드는 인턴이 작성한 보고서와 같습니다. 검토 없이 바로 대표님께 올리진 않죠?
🔒 API 키가 코드에 노출되면?
GitHub Pages에 올리는 HTML은 누구나 소스를 볼 수 있습니다. 결제 API 키가 노출되면 무단 결제, 이메일 키가 노출되면 스팸 발송이 발생합니다.
코드에 넣으면 안 되는 것
결제 API 키 (PG사 시크릿 키)
이메일 발송 API 키
관리자/DB 접속 비밀번호
코드에 있어도 괜찮은 것
Firebase API 키 (Rules로 보호 시)
Google Maps API 키 (도메인 제한 시)
공개 CDN 주소
📜 사용자 입력을 그대로 표시하면?
입력창에 악성 코드를 넣으면 다른 사용자 화면에서 그 코드가 실행됩니다. 누군가 입력창에 "바이러스"를 심으면 접속하는 모든 사람이 감염되는 것입니다.
▼ 코드 예시 보기
// 안전한 사용 element.textContent = userInput; // ✅ 안전 // 위험한 사용 element.innerHTML = userInput; // ❌ 위험!
💻 브라우저에서 로그인 우회가 가능하면?
F12(개발자도구)에서 로그인 함수를 직접 호출하면 비밀번호 없이 접속할 수 있습니다. 주요 함수에 "로그인했는가?" 체크를 추가해야 하고, 서버에서도 검증해야 합니다.
▼ 코드 예시 보기
var _authenticated = false; // 주요 함수에 체크 추가 function renderScreen() { if (!_authenticated) return; // 미인증 시 중단 // ... 화면 렌더링 }
💡 매번 직접 확인하기 어려우시죠?
그래서 CLAUDE.md를 만든 거예요! 첫번째에서 설정해두면 위 검토를 AI가 자동으로 해주거나, "보안 확인해줘" 한마디로 전부 체크합니다.

!
GitHub 리포지토리가 Public으로 되어 있지 않은가?
▼
왜 위험: Public이면 소스코드가 전 세계에 공개됩니다. 코드에 비밀번호나 API 키가 있으면 그대로 노출됩니다.
확인: GitHub → Settings → 하단 Danger Zone에서 Private으로 설정되어 있는지 확인
!
Firebase 콘솔에 보안 규칙이 설정되어 있는가?
▼
왜 위험: allow: true로 되어 있으면 API 키만 아는 사람은 누구든 데이터를 삭제할 수 있습니다.
확인: Firebase 콘솔 → Firestore → 규칙 탭

위험한 규칙이 있으면, 아래 코드로 교체하세요:
Firebase 보안 규칙 (클릭하면 복사)click to copy
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /{document=**} {
      allow read, write: if request.auth != null;
    }
  }
}
이 규칙은 로그인한 사용자만 데이터에 접근할 수 있도록 합니다.
🚨 이 2가지는 개인별·팀별로 즉시 점검해주세요.
발견되면 리드에게 보고 후 즉시 조치해주세요.

🛑
빨간불 — 절대 입력 금지

⚠
노란불 — 주의해서 판단

🔎

✅
초록불 — 자유롭게 가능

02
AI에게 코딩을 시킬 때 이것만 기억하세요
지시하는 방법에 따라 결과의 보안 수준이 완전히 달라집니다
✅ 이렇게 지시하세요
나
로그인 기능 만들어줘. 비밀번호는 해시로 저장하고, 보안 가이드라인 지켜서 구현해줘.
❌ 이렇게 하면 위험해요:
• "빨리 동작만 하게 만들어줘" → 보안이 빠진 코드가 나옵니다
• "보안은 나중에 할게" → 나중은 대부분 오지 않습니다
• "에러 나면 allow: true로 해줘" → 보안이 완전히 해제됩니다
📌 기존 코드를 AI에게 보여줄 때는?
비밀번호나 키가 코드에 있다면 가짜 값으로 바꾼 후 보여주세요.
"sk-abc123실제키..." → 그대로 보여주기 ❌
"YOUR_API_KEY" → 가짜 값으로 바꾼 후 보여주기 ✅
💡 코딩 완료 후: AI에게 "이 코드에서 보안 취약점이 있는지 검토해줘"라고 요청하세요.
