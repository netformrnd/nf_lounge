RULE 01
🔒
비밀번호는 반드시 해시로 저장
평문(Plain Text) 저장 절대 금지
왜? 비밀번호를 그대로 저장하면, 누군가 DB를 열어봤을 때 모든 계정 비밀번호가 그대로 보입니다. 해시로 저장하면 의미 없는 암호문만 보여서 원래 비밀번호를 알 수 없습니다.
🔒비밀번호 원문은 코드에도, DB에도 절대 저장하지 마세요. SHA-256 해시로 변환 후 저장하고, 로그인 시에는 입력값의 해시와 저장된 해시를 비교합니다.
적용 효과: 해커가 DB를 탈취해도 비밀번호 원문을 알 수 없습니다. 관리자도 비밀번호를 볼 수 없어 내부 유출 위험도 차단됩니다. (LinkHub에 적용 완료)
DO
SHA-256 해시로 변환 후 저장
비교는 해시 vs 해시로만
비밀번호 원문은 어디에도 없게
DON'T
HTML/JS에 비밀번호 하드코딩
Firebase에 평문 비밀번호 저장
console.log()로 비밀번호 출력
C
CLAUDE CODE 요청 예시
"비밀번호를 SHA-256 해시로 변환해서 Firebase에 저장하고, 로그인 시 해시 비교로 검증하는 방식으로 구현해줘. 코드에 평문 비밀번호 하드코딩하지 말고."
SHA-256 해시 저장/검증
// SHA-256 해시 변환 함수 async function sha256(str) { const buf = await crypto.subtle.digest( 'SHA-256', new TextEncoder().encode(str) ); return Array.from(new Uint8Array(buf)) .map(b => b.toString(16).padStart(2,'0')) .join(''); } // 저장: 해시만 저장 const hash = await sha256('비밀번호'); db.collection('passwords').doc('teamA') .set({ hash: hash }); // 검증: 해시끼리 비교 const inputHash = await sha256(입력값); if (inputHash === 저장된해시) { // 로그인 성공 }
RULE 02
🛡
Firebase Security Rules 필수 설정
데이터 접근 범위를 반드시 제한
왜? 우리 사이트의 소스코드를 열면 Firebase 접속 정보가 보입니다. 보안 규칙이 없으면 이 정보만으로 누구든 우리 데이터를 몽땅 지워버릴 수 있습니다.
🛡내 서비스가 사용하는 데이터 영역만 허용하고, 나머지는 전부 차단해야 합니다. Firebase 콘솔 > Firestore > 규칙 탭에서 설정합니다.
📌여러 서비스가 같은 Firebase를 쓸 경우, 데이터 이름에 접두사를 붙여 구분하세요. (예: portal-links, hr-employees)
적용 효과: 외부인이 API 키를 알아내도 허용된 데이터만 접근 가능합니다. 다른 서비스의 데이터는 완전히 보호됩니다. 마치 건물 출입증처럼, 특정 층만 출입 가능하게 설정하는 것입니다.
C
CLAUDE CODE 요청 예시
"Firebase Security Rules를 설정해줘. 내 서비스에서 사용하는 컬렉션(예: my-service-links, my-service-settings)만 읽기/쓰기 허용하고, 나머지 컬렉션은 전부 차단해줘."
Security Rules 설정
rules_version = '2'; service cloud.firestore { match /databases/{database}/documents { // 내 서비스 전용 컬렉션만 허용 match /my-service-links/{doc} { allow read, write: if true; } match /my-service-settings/{doc} { allow read, write: if true; } // 나머지는 전부 차단 match /{document=**} { allow read, write: if false; } } }
RULE 03
🔐
API 키 노출 최소화
소스코드에 민감 정보 직접 노출 주의
왜? GitHub Pages에 올리는 HTML은 누구나 소스를 볼 수 있습니다. Firebase API 키는 노출이 불가피하지만, 결제나 메일 발송 같은 API 키가 노출되면 무단 결제나 스팸 발송이 발생할 수 있습니다.
코드에 넣으면 안 되는 것
결제 API 키 (PG사 시크릿 키)
이메일 발송 API 키
관리자/DB 접속 비밀번호
개인정보 (전화번호, 주민번호 등)
코드에 있어도 괜찮은 것
Firebase API 키 (Rules로 보호 시)
Google Maps API 키 (도메인 제한 시)
공개 CDN 주소
서비스 설정값 (색상, 텍스트 등)
C
CLAUDE CODE 요청 예시
"이 코드에서 소스코드에 노출되면 안 되는 민감 정보(API 키, 비밀번호 등)가 있는지 검토해줘. 있으면 안전한 방식으로 대체해줘."
RULE 04
📜
사용자 입력값은 항상 검증
XSS 방지 + 데이터 정합성
왜? 입력창에 악성 코드를 넣으면 다른 사용자 화면에서 그 코드가 실행될 수 있습니다. 쉽게 말해, 누군가 입력창에 "바이러스"를 심으면 접속하는 모든 사람이 감염되는 것입니다.
📜사용자 입력값을 화면에 표시할 때는 반드시 안전하게 변환(이스케이프)해달라고 요청하세요. URL도 올바른 형식인지 검증하세요.
적용 효과: 악의적인 입력이 들어와도 단순 텍스트로 표시되어 피해가 없습니다. 잘못된 URL 입력도 사전에 차단되어 깨진 링크가 등록되지 않습니다.
C
CLAUDE CODE 요청 예시
"사용자 입력값이 HTML에 들어가는 부분에 XSS 방지를 위한 이스케이프 처리를 해줘. URL 입력 필드에는 유효성 검사도 추가해줘."
이스케이프 + URL 검증
// HTML 특수문자 이스케이프 function esc(s) { if (!s) return ''; return s.replace(/&/g, '&amp;') .replace(/</g, '&lt;') .replace(/>/g, '&gt;') .replace(/"/g, '&quot;'); } // URL 유효성 검사 function isValidUrl(s) { try { new URL(s); return true; } catch { return false; } } // 안전한 사용 element.textContent = userInput; // 안전 // 위험한 사용 element.innerHTML = userInput; // 위험!
RULE 05
🗃
Firebase 컬렉션 이름 규칙
추측하기 어려운 이름 사용
왜? Firebase 데이터 이름을 "users", "passwords"처럼 누구나 추측 가능한 이름으로 만들면, 외부에서 해당 이름으로 직접 데이터에 접근을 시도할 수 있습니다.
❌피해야 할 이름: users, passwords, data, settings, admin
✅권장 이름: 서비스명-기능 형태로 접두사 붙이기 (예: portal-pw, hr-employees, kpi-scores)
적용 효과: 데이터 이름만으로는 어떤 서비스의 어떤 데이터인지 외부에서 추측하기 어렵습니다. 같은 Firebase에 여러 서비스를 운영할 때 데이터가 섞이는 것도 방지합니다. (LinkHub 적용 완료: portal-links, portal-pw 등)
C
CLAUDE CODE 요청 예시
"Firebase 컬렉션 이름을 서비스명-기능 형태의 접두사 방식으로 변경해줘. 예를 들어 users 대신 myapp-users 형태로."
RULE 06
💻
개발자도구(F12) 우회 방지
콘솔에서 함수 직접 호출 차단
왜? 브라우저에서 F12를 누르면 개발자도구가 열립니다. 여기서 로그인 함수를 직접 호출하면 비밀번호 없이 접속할 수 있습니다. 주요 함수에 인증 상태 체크를 추가해야 합니다.
🔒로그인 후 실행되는 주요 함수(화면 렌더링, 데이터 추가/수정/삭제)에 "로그인했는가?" 체크를 추가하세요. 로그인하지 않았으면 함수가 실행되지 않게 합니다.
적용 효과: 개발자도구에서 함수를 직접 호출해도, 정상 로그인 없이는 아무 기능도 동작하지 않습니다. 비밀번호를 모르면 접근 자체가 불가합니다. (LinkHub 적용 완료)
C
CLAUDE CODE 요청 예시
"로그인 인증 상태를 변수로 관리하고, 주요 함수(화면 렌더링, 데이터 CRUD, 관리자 기능)에 인증 체크를 추가해줘. 로그인하지 않은 상태에서 콘솔로 함수를 호출해도 동작하지 않게."
인증 체크 패턴
var _authenticated = false; // 로그인 성공 시 function handleLogin() { // 비밀번호 검증 후... _authenticated = true; showMainScreen(); } // 주요 함수에 체크 추가 function renderScreen() { if (!_authenticated) return; // 미인증 시 중단 // ... 화면 렌더링 } // 로그아웃 시 function logout() { _authenticated = false; }
